# 红楼梦 Book Wiki Schema

## 版本

当前版本：**v2.0.0**

| 版本 | 说明 |
|------|------|
| v1.0.0 | 骨架阶段：120回章节摘要 + 42人物 + 32事件 + 20概念 + 16地点 的导航网络 |
| v2.0.0 | 研究型完成：105人物 + 120事件 + 38概念 + 52红学专题 + 12诗词 + 26图片 + 繁简双版本 + 0断链 |

## Domain

本 Wiki 只服务于《红楼梦》的阅读、摘录、解释、人物关系、主题分析、背景知识和红学问题。不要与通用 article/wiki 混合。

## 状态含义

| status | 含义 |
|--------|------|
| `draft` | 只允许临时存在，不能作为最终交付状态 |
| `compiled-v1` | 已有结构化内容，可读但可能仍需人工精修 |
| `curated-v1` | 已去除空壳，有基本解释 |
| `curated-v2` | 已有具体文本判断、章节抓手或命运线 |
| `curated-v3` | 核心页，已按阶段、关系或主题重组 |

## 编译规则

1. `02_Learn/00_Raw/books/红楼梦/` 是原始 source of truth，不可改写；120 回繁体原文已导入 `texts/traditional/`，简体阅读本在 `texts/simplified/`，两者繁简互链。
2. 章节页以“概要 + 结构化索引”为主，不写成长篇赏析。
3. 人物页必须至少包含：人物小传、关键关系、命运线、关键章节。
4. 事件页必须至少包含：事件说明、相关章节、叙事作用、阅读抓手。
5. 概念页必须至少包含：概念说明、如何阅读、关键章节、与其他主题的关系。
6. 诗词页必须至少包含：文本位置、叙事功能、人物/主题关系、可回读的章节入口。
7. 红学页必须至少包含：核心问题、主要观点分歧、证据层级、相关原文/章节。
8. 背景页解释制度文化，不参与剧情复述。
9. 红学争议页必须标记”文本事实 / 研究判断 / 个人理解”的边界。

## 外部研究材料接入规则

红学材料不是孤立堆到 `redology/`，而是按“Wiki 节点 → 外部材料 → 专题综述 → 回链补强”的方式接入。

1. 选题从现有节点出发：优先选择 `characters/`、`concepts/`、`events/`、`poetry/` 中已经可读、且阅读时容易产生分歧的问题。
2. 先补节点，再建专题：人物、概念、事件页中保留简短研究提示；复杂争议集中到 `redology/` 专题页。
3. 外部引用必须落在相应页面 frontmatter 的 `sources:` 字段；如果某个页面吸收了外部观点，该页面自己也要标注来源，不只在总综述页标注。
4. `redology/` 专题页必须说明研究范围、代表材料、主要分歧、可对读章节和相关 Wiki 节点。
5. 同一问题允许保留互相冲突的观点，但要区分“原文事实”“学者判断”“本 Wiki 的阅读建议”。
6. 外部材料只作为研究入口，不替代原文；所有结论必须能回链到章节、人物、概念或事件。
7. 每轮研究完成后，同步更新 `index.md`、`outputs/红楼梦主题导读.md`、`outputs/红楼梦编译总评.md` 和 `log.md`。

### 推荐工作流

1. 从一个现有节点开始，例如 [[02_Learn/08_book-wikis/红楼梦/characters/王熙凤.md]] 或 [[02_Learn/08_book-wikis/红楼梦/concepts/金玉良缘.md]]。
2. 搜索权威材料：学术论文、出版社书目信息、研究机构资料、作家网/高校期刊/权威媒体文章。
3. 在原节点补一段“研究视角”，并把外部链接写入该节点 `sources:`。
4. 若材料已经形成争议网络，再创建一个 `redology/xxx研究综述.md`。
5. 从专题页回链到相关人物、章节、事件、概念、诗词。
6. 运行链接和低信息量检查，保证没有断链和空壳页。

## 禁止的维护方式

不要再创建只有标题和空泛说明的页面。以下句式视为质量问题：

- “作为关键事件入口保留”
- “推动人物关系变化”
- “通过相关章节可以追踪”
- “本回未单列”
- “待补 / TODO / 暂无”
- 只把回目拆成两句话，没有任何解释判断

如果抽取到的人物、事件或概念暂时无法写出实质说明：

1. 不创建独立页；
2. 在章节页中保留纯文本；
3. 等能写出小传、叙事作用或主题关系后再升级为页面。

## 链接规则

- 章节链接：`[[02_Learn/08_book-wikis/红楼梦/chapters/第001回.md]]`
- 人物链接：`[[02_Learn/08_book-wikis/红楼梦/characters/林黛玉.md]]`
- 事件链接：`[[02_Learn/08_book-wikis/红楼梦/events/宝玉挨打.md]]`
- 原文链接：优先链接到 `texts/traditional/`（繁体）或 `texts/simplified/`（简体）中的对应章节。

## 原文锚点规则

当人物、事件、概念需要跳到章节内部的具体原文位置时，优先使用简体原文页中的 Obsidian block anchor。

- 事件页必须包含“原文锚点”小节，链接到简体阅读层的 block id。
- 人物页、概念页后续补强时，也优先补入同类原文锚点。
- Raw 繁体原文仍然只读，不插入锚点；Raw 只作为来源层保存。
- 简体原文 `texts/simplified/` 可以在关键段落末尾加入稳定 block id，例如：`^hlm-008-baochai-kan-yu`。
- 事件页、人物页、概念页用 `[[02_Learn/08_book-wikis/红楼梦/texts/simplified/第008回.md#^hlm-008-baochai-kan-yu|第008回原文：宝钗看玉]]` 链接到具体段落。
- block id 命名规则：`^hlm-回数-主题短名`，例如 `^hlm-096-diaobao-plan`。
- 多个页面指向同一段原文时，可以共用同一个语义锚点，避免在同一段落堆叠多个 block id。
- 不建议使用行号作为长期链接，因为行号会随编辑变化；不建议在 Raw 原文中插入标题，以免污染 source of truth。

## 表格链接规则

- 表格内如需使用带别名 Wiki 链接，必须转义别名分隔符：`［［path\|别名］］`，不要写成 `［［path|别名］］`，否则 Markdown 表格会把 `|` 当作列分隔符。

## 批量更新后的检查

每次批量更新后至少检查：

1. Wiki 双链目标缺失数为 0。
2. 非日志文件中不存在禁止句式。
3. 新建页面不是低信息量薄页。
4. `outputs/红楼梦编译总评.md` 的数量和质量判断同步更新。
5. `log.md` 记录本次改动范围。
6. output 型导览页直接链接到的子页面（redology / poetry / concepts / events / characters）不能只有一两句占位说明；达不到质量门槛的不能包装成成熟入口。
